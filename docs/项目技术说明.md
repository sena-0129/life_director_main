### 2.1 内容补全（RAG）如何实现

目标：用户在“整理完成页”点击“基于记忆补全”，系统会从同一用户的历史故事中检索最相关的几条，并生成一段更完整的故事文本。

下面按“数据进入系统 → 建索引 → 检索 → 生成 → 回填”的顺序，把端到端工作流展开说明。

#### 数据写入阶段（为检索做准备）

当用户保存故事（“保存到博物馆”）时：

1) 后端接收 `content`（故事正文）
2) 调用 DashScope embedding 接口生成向量（1024 维）
3) 写入 Supabase：`stories` 表
   - `content`：原文
   - `embedding`：向量（`vector(1024)`）
   - `owner_key`：用于“同一设备用户隔离”

更细的时序（本地开发为例）：

1) 用户在“故事确认页”点击“保存到博物馆”
2) 前端调用 `storage.saveStory(story)`
   - 如果 `VITE_USE_BACKEND=true`：走后端 API（而不是 localStorage）
   - 请求路径：
     - 新故事：`POST /api/profiles/:profileId/stories`
     - 更新故事：`PUT /api/stories/:id`
   - 请求头：
     - `Authorization: Bearer <VITE_BACKEND_TOKEN>`（可选）
     - `x-user-key: <life_director_user_key_v1>`（必带，用于隔离）
3) 后端落库前先生成 embedding：
   - 调用 DashScope OpenAI 兼容 `/embeddings`
   - model：`text-embedding-v4`
   - dimensions：`1024`
4) 后端插入/更新 `public.stories`
   - `embedding` 字段写入 `vector(1024)`
   - `owner_key` 写入请求的 `x-user-key`

关键实现点：
- 生成 embedding 并写入：见 `POST /api/profiles/:id/stories`、`PUT /api/stories/:id`
  - 本地后端：[server/index.ts](file:///Users/simiao/life-director-main/server/index.ts)
  - 线上后端：[api/index.ts](file:///Users/simiao/life-director-main/api/index.ts)
- 前端保存调用：`storage.saveStory`，见 [storage.ts](file:///Users/simiao/life-director-main/src/utils/storage.ts)

注意：如果 Supabase 未启用（本地后端未配置 `SUPABASE_*`），会退化为 SQLite 存储，此时 RAG 检索能力在本项目实现里要求 Supabase（`/api/rag/run` 会直接拒绝）。

实现位置：
- 本地后端写入逻辑：[server/index.ts](file:///Users/simiao/life-director-main/server/index.ts)
- 线上后端写入逻辑：[api/index.ts](file:///Users/simiao/life-director-main/api/index.ts)

#### 检索 + 生成阶段（RAG run）

前端点击“基于记忆补全”后：

1) 前端调用：`POST /api/rag/run`
   - body：`{ userId, userInput, topK }`
   - header：
     - `Authorization: Bearer <token>`（如果启用了 `BACKEND_TOKEN`）
     - `x-user-key: <uuid>`（用于隔离用户数据）
2) 后端对 `userInput` 生成 embedding
3) 后端拉取该用户历史 stories（同 `profile_id`、同 `owner_key`）
4) 在后端做相似度计算（cosine similarity），选 TopK
5) 拼接 prompt（包含“用户输入 + 相关记忆”）
6) 调用 DashScope chat completion 生成增强文本
7) 返回：`{ enhanced, relatedStories }`，前端回填到文本框

更细的时序（把每一步的输入输出讲清楚）：

**A. 前端发起补全请求**

1) 用户在“整理完成页”点击“基于记忆补全”
2) 前端取当前档案 `profileId` 作为 `userId`（这里的 `userId` 实际是档案 id）
3) 前端取当前文本框内容作为 `userInput`
4) 调用 `POST /api/rag/run`
   - body：`{ userId, userInput, topK: 3 }`
   - headers：
     - `Authorization: Bearer <VITE_BACKEND_TOKEN>`（可选）
     - `x-user-key: <life_director_user_key_v1>`（必带）

**B. 后端检索 TopK 相关记忆**

5) 后端生成查询向量：对 `userInput` 调 DashScope embedding
6) 后端读取候选集合：
   - 先校验该 `profileId` 必须属于当前 `owner_key`（防止越权读取）
   - 再查询 `stories`：条件为 `profile_id = userId AND owner_key = x-user-key`
7) 后端在应用层计算余弦相似度：
   - `cosineSimilarity(queryEmbedding, story.embedding)`
   - 排序并截取 topK
   - 注意：这里不是走 pgvector 的 `ivfflat`/`<->` SQL 相似度检索，而是后端把候选故事拉出来在内存里算（实现简单，数据量大时需要优化）

**C. 构造 Prompt 并生成增强文本**

8) 后端用 `buildRagPrompt(userInput, topKStories)` 拼 prompt，通常包含：
   - 用户输入原文
   - 相关记忆列表（每条包含 year/tags/content）
   - 风格与约束（温暖、克制、不编造、第一人称等）
9) 后端调用 DashScope chat completion（Qwen），得到 `enhanced`

**D. 回传结果并回填 UI**

10) 后端返回 JSON：
   - `enhanced`：补全后的故事
   - `relatedStories`：本次命中的候选记忆及相似度
11) 前端将 `enhanced` 回填到编辑框，同时展示“相关记忆”列表，便于用户理解 AI 参考了什么

关键实现点：
- 前端入口按钮与回填： [StoryConfirmPage.tsx](file:///Users/simiao/life-director-main/src/pages/StoryConfirmPage.tsx)
- 前端请求封装： [rag.ts](file:///Users/simiao/life-director-main/src/services/rag.ts)
- 后端检索 + prompt + 生成：
  - 本地后端：`POST /api/rag/run` + `buildRagPrompt`，见 [server/index.ts](file:///Users/simiao/life-director-main/server/index.ts)
  - 线上后端：同名实现，见 [api/index.ts](file:///Users/simiao/life-director-main/api/index.ts)

常见故障点（排查顺序）：
- 前端报 `Failed to fetch`：通常是本地 `/api` 代理未连上后端（Vite proxy 或后端未启动）
- 前端报 `unauthorized`：后端开了 `BACKEND_TOKEN`，但前端没带 `VITE_BACKEND_TOKEN`
- 后端返回 `profile not found`：传入的 `userId/profileId` 不属于当前 `x-user-key`（隔离生效）
- 补全文本仍出现第二人称：需要同时修改本地后端与 Vercel 后端的 prompt/system message（两套入口）

实现位置：
- `POST /api/rag/run`：[server/index.ts](file:///Users/simiao/life-director-main/server/index.ts)、[api/index.ts](file:///Users/simiao/life-director-main/api/index.ts)
- Prompt 拼接函数：`buildRagPrompt(...)`
- 第一人称强约束：系统消息（system message）与 prompt 都有约束

前端调用位置：
- “整理完成页”按钮触发： [StoryConfirmPage.tsx](file:///Users/simiao/life-director-main/src/pages/StoryConfirmPage.tsx)
- 请求封装： [rag.ts](file:///Users/simiao/life-director-main/src/services/rag.ts)

### 2.2 人生短片生成如何实现（豆包两段式工作流）

目标：在“人生视频工作台”选择多条故事素材，点击“生成我的人生短片”，系统先生成“脚本/提示词”，再调用视频模型生成 mp4，并保存到 Supabase Storage。

下面按“素材选择 → 脚本生成 → 视频任务创建 → 轮询完成 → 上传存储 → 找回播放”的端到端顺序展开。

#### 前端工作流

1) 用户在“人生视频工作台”选择故事卡片
2) 点击“生成我的人生短片”
3) 前端把选中的故事 `content` 拼接为一个长文本作为素材
4) 调用后端：`POST /api/ai/video`
5) 后端返回 `{ videoUrl: "/api/ai/video/<id>" }`
6) 前端随后请求 `GET /api/ai/video/<id>`：
   - 如果还在生成：后端返回 `202 { status }`，前端轮询
   - 生成完成：后端返回 mp4（通过 redirect 到 Supabase signedUrl），前端下载成 blob 并播放/允许下载

补充两点前端实现细节：

- **鉴权与隔离**：前端请求会自动带上
  - `Authorization: Bearer <VITE_BACKEND_TOKEN>`（可选）
  - `x-user-key: <life_director_user_key_v1>`（用于“我的视频列表”隔离）
- **轮询策略**：当 `GET /api/ai/video/:id` 返回 202 时，前端每 5 秒重试一次，最长等待约 25 分钟（防止无限等待）

实现位置：
- 页面： [VideoStudioPage.tsx](file:///Users/simiao/life-director-main/src/pages/VideoStudioPage.tsx)
- 生成请求封装： [gemini.ts](file:///Users/simiao/life-director-main/src/services/gemini.ts) 的 `generateLifeVideo()`（名称沿用，但后端已切到豆包）

#### 后端工作流（脚本生成 → 视频生成 → 入库）

当后端接到 `POST /api/ai/video`：

1) 脚本生成（Seed 模型）：
   - 调用 Ark `chat/completions`
   - 使用 `doubao-seed-2-0-lite-260215` 将“故事素材”改写成更适合文生视频的镜头脚本/提示词
2) 视频生成（Seedance 模型）：
   - 调用 Ark `contents/generations/tasks` 创建任务
   - 使用 `doubao-seedance-1-5-pro-251215`
3) 先写入 `ai_videos`：保存 `ark_task_id`、`status`、`script`、`prompt`，并返回 `videoUrl` 给前端
4) 当 `GET /api/ai/video/:id` 被访问时：
   - 轮询 Ark `contents/generations/tasks/:taskId`
   - 成功后下载视频二进制
   - 上传到 Supabase Storage 的 `videos` bucket（文件名：`<id>.mp4`）
   - 更新 `ai_videos.object_path`
   - 返回 redirect 到 signed url

更细的时序（含存储与状态字段）：

**A. 创建任务（POST /api/ai/video）**

1) 后端读取 `prompt`（这里的 prompt 是“故事素材拼接文本”，还不是最终视频提示词）
2) 调用脚本模型（Seed）生成“镜头脚本/提示词”：
   - 目标是让提示词更“视频友好”（画面、动作、镜头、光线、年代氛围）
   - 同时约束：不出现“你/您”，避免把读者当作当事人
3) 将脚本拼成最终 T2V prompt（例如附带分辨率、时长、比例参数）
4) 调用视频模型（Seedance）创建异步任务，得到 `taskId`
5) 写入 Supabase `ai_videos`（此时还没有 mp4 文件）：
   - `id`：本项目生成的 uuid（用于对外 videoUrl）
   - `owner_key`：来自 `x-user-key`
   - `ark_task_id`：视频平台任务 id
   - `status`：初始状态（通常是 queued/processing 一类）
   - `script`：脚本模型输出
   - `prompt`：最终喂给视频模型的提示词
   - `bucket`：固定 `videos`
   - `object_path`：暂为空（表示“尚未落到 Storage”）
6) 返回给前端：`{ videoUrl: "/api/ai/video/<id>" }`

**B. 取回视频（GET /api/ai/video/:id）**

7) 前端请求 `GET /api/ai/video/:id`
8) 后端读取 `ai_videos` 记录：
   - 如果 `object_path` 已存在：直接 `createSignedUrl` 并 `302 redirect` 到 mp4
   - 如果 `object_path` 为空：说明文件还没落到 Storage，需要继续查询平台任务
9) 后端轮询 Ark：`GET /contents/generations/tasks/:taskId`
   - 若未完成：返回 `202 { status }` 给前端（让前端继续轮询）
   - 若失败：返回 500
   - 若成功：从任务结果里解析出视频下载 URL
10) 后端下载视频二进制（mp4）并上传到 Supabase Storage：
   - bucket：`videos`
   - object_path：`<id>.mp4`
11) 更新 `ai_videos.object_path`（此后就可直接重定向播放）
12) 返回 `302 redirect` 到 signedUrl

关键实现点：
- Ark API 调用封装： [ark.ts](file:///Users/simiao/life-director-main/server/ark.ts)
- 视频接口：
  - 本地后端：`POST /api/ai/video`、`GET /api/ai/video/:id`，见 [server/index.ts](file:///Users/simiao/life-director-main/server/index.ts)
  - 线上后端：同名路由，见 [api/index.ts](file:///Users/simiao/life-director-main/api/index.ts)

#### 为什么“生成了但 videos bucket 没看到”

关键点：视频文件写入 `videos` bucket 的动作发生在 **拉取/播放/下载阶段**（`GET /api/ai/video/:id`），而不是 `POST /api/ai/video` 立刻写入。

因此：
- 如果你只点了生成、但从未触发后续 `GET /api/ai/video/:id`，bucket 里可能暂时看不到 mp4。
- 为了解决“用户当时不下载就找不到”的体验，本项目在视频工作台新增“我的视频列表”，用户可以事后再点“播放/下载”触发补写入。

另外一个容易忽略的点：
- `videos` bucket 是 **私有 bucket**（`public=false`），你在控制台里看到了文件也无法直接公开访问；应用侧通过 `createSignedUrl` 临时签名进行播放/下载。

## 3. 关键环境变量（与两大核心功能相关）

内容补全（DashScope）：
- `DASHSCOPE_API_KEY`
- `DASHSCOPE_BASE_URL`（默认 `https://dashscope.aliyuncs.com/compatible-mode/v1`）
- `DASHSCOPE_EMBEDDING_MODEL`（`text-embedding-v4`）
- `DASHSCOPE_EMBEDDING_DIMENSIONS`（`1024`）
- `DASHSCOPE_CHAT_MODEL`（`qwen3.5-plus`）

视频生成（火山方舟 Ark）：
- `ARK_API_KEY`
- `ARK_BASE_URL`（默认 `https://ark.cn-beijing.volces.com/api/v3`）
- `ARK_SCRIPT_MODEL`（`doubao-seed-2-0-lite-260215`）
- `ARK_VIDEO_MODEL`（`doubao-seedance-1-5-pro-251215`）

Supabase（后端用）：
- `SUPABASE_URL`
- `SUPABASE_SERVICE_ROLE_KEY`

后端鉴权（可选，但生产环境建议开启）：
- `BACKEND_TOKEN`
- `VITE_BACKEND_TOKEN`
